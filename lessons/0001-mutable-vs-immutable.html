Mutable vs. Immutable: the split that reshapes your project

The single change that reorganizes your whole codebase. Get it wrong and your Cloud Manager build fails before a line of your code runs. Get it right and everything downstream falls into place.

The core idea

On classic AEM, you deploy one content package and it can write anywhere in the repository. On AEM as a Cloud Service, the repository is split in two, and code and content can no longer live in the same package.

The rule Immutable = code. Written only at deploy time, read-only at runtime: /apps and /libs.
Mutable = everything a running author instance can change: /content, /conf, /home, /var, /etc, /oak:index.

Why the hard line? Cloud Service runs many replaceable instances that spin up from a golden image built by the pipeline. Immutable code is baked into that image so every instance is identical. Mutable content lives in the shared repository the instances attach to. Mixing them means a deploy would either fail or silently clobber author changes.

What it means for your package structure

Your Maven project must produce a container package that embeds two sub-packages — Adobe's tooling enforces this split at build time:

Classic AEM — one package

ui.apps/
  /apps/myco/...
  /content/myco/...   ← mixed in
  /conf/myco/...      ← mixed in

Cloud Service — split

all/            (container)
 ├─ ui.apps/    → /apps  (IMMUTABLE)
 └─ ui.content/ → /content, /conf
                  (MUTABLE)

The ui.apps package is marked <packageType>application</packageType> and may contain only immutable paths. If a single node under /content sneaks in, the package structure validation fails the pipeline.

Gotcha from real migrations /oak:index definitions are mutable but ship as part of your project. And /apps is read-only at runtime — so any code that tried to write to /apps on a running instance (some legacy config patterns did) must be refactored, often to RepoInit.
